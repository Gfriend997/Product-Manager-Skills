## Recipe: Add One More Skill (for the sibling(s))

Follow the same path that produced `incoming-request-advisor`. Dogfood the repo's own scripts over judgment.

1. **Identify type** — component (artifact), interactive (adaptive Q&A + numbered options), or workflow (orchestration). Inbound-message / response skills are almost always **interactive**.
2. **Create `skills/<name>/SKILL.md`** with:
   - Frontmatter: `name`, `argument-hint`, `description` (**≤200 chars, trigger-oriented, quote it if it contains a colon**), `intent` (`>-` block), `type`, `theme`, `best_for`, `scenarios`, `estimated_time`.
   - The 7 sections **in order**: Purpose, Input, Key Concepts, Application, Examples, Common Pitfalls, References.
   - Input section as **invitation, not gate**: `**Works best with:**` → `**Also useful:**` → the inline-input rule (invocation context = answers already given, don't re-ask) → `**Arriving empty-handed? That works too.**` → an example invocation. Never label anything "Required."
   - For interactive skills: add the `### Facilitation Source of Truth` block linking `../workshop-facilitation/SKILL.md`, and register the skill in `docs/maintenance/facilitation-scope.md`'s scope snapshot.
3. **Add `template.md`** if the skill produces a reusable artifact: `# <X> Template` → `## Provenance` (cite the `product-manager-prompts` source prompt + Dean attribution) → `## Template` fenced block. Link it from the Application section.
4. **Add `examples/conversation-flow.md`** for interactive skills (anonymized end-to-end run).
5. **Add the marketplace entry** — see the encoding trap in Gotchas. Insert only the new object into the `plugins` array; keep it sorted by `name`; `description` is the trigger-free first sentence; `category` from the existing set (`ai, career, delivery, discovery, finance, meta, stakeholder, strategy`); 4–5 `tags`; `strict: false`.
6. **Do NOT touch cross-references to plugin-only skills.** `ps-*` skills (e.g. `ps-jobs-to-be-done`) live in the marketplace/plugin bundle, **not** in this repo's `skills/`. Linking them from a SKILL.md breaks `check-library-drift.py`. Link only skills that have a `skills/<name>/` directory here.
7. **Validate** (all must pass, 0 new warnings):
   ```bash
   python3 scripts/check-skill-metadata.py skills/<name>/SKILL.md
   bash scripts/test-a-skill.sh --smoke <name>
   python3 scripts/check-library-drift.py
   ```

## Gotchas (specific to this work)

- **`marketplace.json` encoding trap — this bit us on `incoming-request-advisor`.** A naive `json.dump(..., ensure_ascii=False)` rewrites every `—` escape in the file to a literal em-dash, producing a huge diff across unrelated entries. **Use `json.dumps(d, indent=2, ensure_ascii=True)` (the default), preserve the trailing newline, and insert only your new object.** Always `git diff` the file and confirm the diff shows *only* your entry before committing.
- **Counts are deliberately stale on `main` right now.** If you see 55 in the README while `check-library-drift.py` reports 56, that is the expected in-between state, not a bug to hotfix. Fix it as part of the release bump, all surfaces at once.
- **`marketplace.json` is hand-maintained.** No script adds entries; drift only catches a *missing* entry, not a malformed count.
- **The ASCII banner in README (line ~17) is width-sensitive.** 55→56/57/58 are all two digits, so the box stays aligned; if a count ever crosses to three digits, re-pad the box.
- **Bulk skill edits:** use a Python script via Bash, not the Edit tool — files drift between reads. Quote YAML `description` values containing colons.
- **Source prompts live in a sibling repo:** `https://github.com/deanpeters/product-manager-prompts`. The `incoming-request-breakdown.md` prompt was at `prompts/` there. Expect the siblings to come from the same place; cite them in Provenance.
